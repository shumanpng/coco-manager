## Claim / Question / Decision

Storing each state/planning item as its own file with a generated `INDEX.md` table is essential to the system's scalability. When agents scan for existing items (e.g., to dedupe), reading only the index — not all items — keeps token consumption manageable as the knowledge base grows. This emerged organically from author use, has been refined through iterations, and is a load-bearing design choice for both performance and human navigation.

## Evidence

- Worklog:
- Meeting minutes:
- Artifacts: `state/INDEX.md`, `planning/INDEX.md`
- Code pointers: `scripts/generate_state_index.py`, `scripts/generate_planning_index.py`
- Commands run: `make state-index`, `make planning-index`

## What would change my mind

A simpler structure (e.g., one giant file per type) demonstrably scales as well at 100+ items without context-window strain. Or: agents start ignoring the index and reading all items anyway, making the optimization moot.

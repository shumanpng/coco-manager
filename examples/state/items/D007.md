## Claim / Question / Decision

End-session and process-meeting agents must always read `INDEX.md` first (one file), then open individual item files only for IDs with title/tag overlap with the candidate — never read all items upfront.

**Rationale:** Eliminates O(N) item-file reads (26+ planning, 63+ state items) per session; reduces to O(k) where k is typically 0–5.

**Consequences:**
- INDEX must be up-to-date: generation scripts must run after each update session (already required by the post-update consistency check)
- Semantic matching via title/tags may occasionally miss a closely-related item phrased very differently — instructions say "if uncertain, read the full item"; acceptable risk given the token savings

## What would change my mind

If selective reads consistently miss duplicate items in practice (i.e., multiple sessions produce duplicates due to missed title/tag overlap), the protocol should be revised to include a broader keyword-match pass or a required human review step before writing.
